MASTER BUILD PROMPT — Trading Desk Copilot: Evidence-Based AI Day-Trading Research Terminal

Role

You are a senior AI systems engineer, Python architect, trading-systems designer, and reliability-focused developer working inside the existing trading-desk-copilot project.

Your task is to build the full next-generation version of this project:

An evidence-based, replay-capable, journal-validated, multi-agent AI day-trading copilot and research terminal.

This is not an auto-trading bot.

The core rule is:

Deterministic code is the source of truth. LLM agents may analyze, debate, critique, explain, and summarize only. They must never place trades, approve trades, submit orders, simulate exchange execution, or override hard risk blocks.

Use TauricResearch/TradingAgents only as an architectural reference for multi-agent role separation, bull/bear debate, risk criticism, final synthesis, memory, provider abstraction, and future checkpointing.

Do not import TradingAgents as a dependency.
Do not copy the repo wholesale.
Do not add broker execution.
Do not add simulated exchange execution.

⸻

1. Product Definition

Build:

Trading Desk Copilot — Evidence-Based AI Trading Research Terminal

The system should support:

1. Live Mode
    * Monitors live market data.
    * Detects deterministic trigger stacks.
    * Shows dashboard alerts.
    * Runs analyst committee.
    * Tracks manual positions.
    * Journals manual outcomes.
2. Replay Mode
    * Loads a historical trading day.
    * Replays it through the same copilot pipeline as if it were live.
    * Allows manual practice and journaling.
    * Marks all results as REPLAY.
3. Strategy Lab
    * Separates primary edge hypotheses from entry-refinement features.
    * Measures whether trigger stacks have positive expectancy.
    * Prevents folklore concepts from being treated as proven alpha.
4. Multi-Agent Analyst Committee
    * TradingAgents-inspired analyst layer.
    * Technical, Pattern, Regime, Order-Flow, Catalyst, Position, Memory, Bull Case, Bear Case, Risk Critic, Final Synthesizer.
    * Reads structured copilot events only.
    * Produces dashboard-safe recommendations.
5. Journal + Validation Loop
    * Records manual decisions and outcomes.
    * Tracks result in R.
    * Feeds results back into validation.
    * Promotes only measured edges.
    * Prevents fake confidence from thin samples.
6. Dashboard
    * Live Board.
    * Replay Panel.
    * Position Panel.
    * Analyst Committee Panel.
    * Risk Gate Panel.
    * Edge Scoreboard.
    * Journal Panel.
    * Feed Quality Panel.
    * History Log.

10.11 Final Synthesizer

File:

agents/final_synthesizer.py

Combines all subreads into one dashboard-safe read.

Output:

{
  "agent": "final_synthesizer",
  "status": "OK",
  "one_sentence_read": "string",
  "final_recommendation": "WATCH | WAIT | AVOID | POSSIBLE_LONG_ZONE | POSSIBLE_SHORT_ZONE | THESIS_VALID | THESIS_WEAKENING | TRAIL_STOP | TAKE_PARTIALS | EXIT_WARNING | THESIS_INVALIDATED | DO_NOT_ADD",
  "confidence": 0.0,
  "what_supports": [],
  "what_argues_against": [],
  "what_confirms": [],
  "what_invalidates": [],
  "position_guidance": [],
  "risk_notes": []
}

Rules:

* Must never override deterministic alert level.
* Must never override L5 block.
* Must obey risk_critic.max_allowed_recommendation.
* Must use only approved recommendation enum.
* Must not include forbidden language.
* Must not invent facts.
* Must not imply certainty.

Final recommendation priority:

1. L5 / hard block → AVOID, DO_NOT_ADD, EXIT_WARNING, or THESIS_INVALIDATED only.
2. In-position thesis invalidated → THESIS_INVALIDATED or EXIT_WARNING.
3. In-position thesis weakening → THESIS_WEAKENING, TRAIL_STOP, DO_NOT_ADD, or EXIT_WARNING.
4. Flat with blocked/weak setup → AVOID or WAIT.
5. Flat with incomplete confirmation → WATCH or WAIT.
6. Flat with confirmed setup and acceptable gates → POSSIBLE_LONG_ZONE or POSSIBLE_SHORT_ZONE.

The final synthesizer must obey this priority order even if the bull case is strong.

13. Strategy Lab

Create:

strategy_lab.py

Goal:

Create a research layer that separates primary edge hypotheses from intraday entry-refinement features.

Definitions:

Primary edge hypothesis = documented or testable market anomaly/context that may create positive expectancy.
Entry refinement feature = intraday trigger used only to time or filter an entry.
LLM analyst = explanation layer only.

Initial strategies:

POST_EARNINGS_DRIFT
RELATIVE_STRENGTH_MOMENTUM

20. L5 Block Rule

Non-negotiable.

If event includes:

* L5 block
* deterministic block
* quote/spread block
* stale quote block
* hard risk block
* data failure block

Then:

* Risk critic must output risk_verdict: "BLOCK".
* Final recommendation must be one of:
    * AVOID
    * DO_NOT_ADD
    * THESIS_INVALIDATED
    * EXIT_WARNING

No agent may soften this into:

* WATCH
* WAIT
* POSSIBLE_LONG_ZONE
* POSSIBLE_SHORT_ZONE
* THESIS_VALID

Add explicit tests.

24. Final Product Vision

The final system should be:

A retail evidence-based AI trading research terminal.

Not:

An autonomous trading bot.

The strongest architecture is:

deterministic copilot event
+ live/replay data modes
+ TradingAgents-style analyst committee
+ QuantAgent-style intraday reads
+ risk critic
+ Strategy Lab
+ Edge Scoreboard
+ journal validation
+ no execution

The system’s purpose is to:

* reduce impulsive trading
* expose weak setups
* track position thesis
* measure whether trigger stacks work
* separate folklore from measured edge
* provide replay practice
* build personal evidence from journaled outcomes
* improve decision discipline

The system must not pretend that LLM analysis equals alpha.

Rule:

No measured edge, no trust.
